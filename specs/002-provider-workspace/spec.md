# Feature Specification: Provider and Workspace Setup

**Feature Branch**: `N/A`

**Created**: 2026-07-04

**Status**: Draft

**Input**: User description — multi-provider support (OpenAI, Anthropic) with
custom URL via .env, workspace directory structure, and Claude-like formatting
configuration.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Run tasks with OpenAI and Anthropic providers (Priority: P1)

An SRE engineer configures minirun to use OpenAI or Anthropic as the LLM
provider. The runtime loads the selected provider, authenticates with the
configured credentials, and executes tasks exactly as it would with the default
provider.

**Why this priority**: Provider flexibility is the core of this feature. Users
must be able to choose between providers based on availability, cost, or
capability requirements. Without this, the runtime is locked to a single LLM
backend.

**Independent Test**: Can be fully tested by configuring minirun with an OpenAI
API key, running a task, then switching to an Anthropic API key and running the
same task, observing equivalent behavior from both.

**Acceptance Scenarios**:

1. **Given** a valid OpenAI API key is configured, **When** the user runs
   `minirun @sre analyze "summarize this incident"`, **Then** the runtime
   returns a response generated by the OpenAI model.
2. **Given** a valid Anthropic API key is configured, **When** the user runs
   the same task, **Then** the runtime returns a response generated by the
   Anthropic model.
3. **Given** both providers are configured with valid keys, **When** the user
   specifies a provider in the profile or command, **Then** the runtime uses
   the specified provider.

---

### User Story 2 — Use a custom API URL for a provider (Priority: P1)

An SRE engineer who uses a self-hosted LLM gateway or API proxy configures a
custom base URL for a provider via environment variables. The runtime uses the
custom URL instead of the default API endpoint.

**Why this priority**: Many organizations route LLM traffic through internal
gateways for security, cost tracking, or compliance. Without custom URL
support, the tool cannot be adopted in these environments.

**Independent Test**: Can be fully tested by setting a custom base URL via .env
that points to an API proxy, then running a task and verifying the request
reaches the proxy endpoint instead of the default provider URL.

**Acceptance Scenarios**:

1. **Given** a custom API URL is set in `.env` (e.g.,
   `OPENAI_BASE_URL=https://gateway.internal.company.com/v1`), **When** the
   user runs a task, **Then** the runtime sends the request to the custom URL
   instead of the default OpenAI endpoint.
2. **Given** no custom URL is configured, **When** the user runs a task, **Then**
   the runtime uses the default API endpoint for the selected provider.

---

### User Story 3 — Workspace directory with organized subfolders (Priority: P2)

A developer working with minirun finds the project automatically creates and
uses a `workspace/` directory with four standard subdirectories: `memory/`,
`agents/`, `commands/`, and `skills/`. This mirrors the familiar convention
from Claude Code / OpenCode projects.

**Why this priority**: A well-known directory structure reduces friction for
users coming from similar tools and provides a predictable location for user
content, profiles, and extensions.

**Independent Test**: Can be fully tested by initializing minirun in an empty
directory and verifying the `workspace/` tree is created automatically with the
four subdirectories.

**Acceptance Scenarios**:

1. **Given** an empty project directory, **When** minirun is initialized,
   **Then** the `workspace/` directory is created with `memory/`, `agents/`,
   `commands/`, and `skills/` subdirectories.
2. **Given** a profile file placed in `workspace/agents/`, **When** the user
   runs a task referencing that profile, **Then** the runtime discovers and
   loads the profile from the workspace directory.
3. **Given** a skill placed in `workspace/skills/`, **When** the runtime loads
   skills, **Then** it discovers the skill from the workspace directory.

---

### Edge Cases

- What happens when both providers are configured but neither has a valid API
  key? The runtime should return a clear error identifying which provider
  failed authentication.
- What happens when a custom URL is set but unreachable? The runtime should
  return a timeout or connection error message — not hang indefinitely.
- What happens when the workspace directory already exists with content?
  minirun should use the existing directory without overwriting or modifying
  any files.
- How does the system handle conflicting configuration between .env and
  settings.yaml? The .env values should take precedence for secrets, with
  settings.yaml providing defaults.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Users MUST be able to configure and use OpenAI as an LLM
  provider.
- **FR-002**: Users MUST be able to configure and use Anthropic as an LLM
  provider.
- **FR-003**: Users MUST be able to specify a custom base URL for any provider
  via environment variables in `.env`.
- **FR-004**: The runtime MUST fall back to the provider's default API endpoint
  when no custom URL is configured.
- **FR-005**: Upon initialization, minirun MUST create a `workspace/` directory
  with `memory/`, `agents/`, `commands/`, and `skills/` subdirectories if they
  do not already exist.
- **FR-006**: The runtime MUST discover profiles from `workspace/agents/` in
  addition to the built-in `profiles/` directory.
- **FR-007**: The runtime MUST discover skills from `workspace/skills/` and
  commands from `workspace/commands/`.
- **FR-008**: Configuration via `.env` MUST take precedence over
  `config/settings.yaml` for all provider-related settings (API keys, base
  URLs).

### Key Entities *(include if feature involves data)*

- **Provider**: An LLM backend (OpenAI, Anthropic) implementing the
  `provider.complete(messages, tools)` interface. Configured via API key +
  optional base URL.
- **Workspace**: A project-level directory (`workspace/`) that serves as the
  user's working area within minirun, organized into standard subdirectories.
- **Agent Profile**: A YAML/Markdown configuration file stored either in
  `profiles/` (built-in) or `workspace/agents/` (user-created) that defines a
  profile's name, system prompt, and allowed tools.
- **Skill**: An extension or capability stored in `workspace/skills/` that adds
  new behavior to the runtime.
- **Command**: A user-defined CLI command stored in `workspace/commands/`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can configure OpenAI as the provider and run a task in
  under 2 minutes from a fresh project setup.
- **SC-002**: A user can switch from the default provider endpoint to a custom
  URL by adding a single line to `.env`, and the next task invocation uses the
  custom endpoint.
- **SC-003**: Running `minirun init` (or the first run equivalent) creates the
  full `workspace/` directory tree within 1 second.
- **SC-004**: A profile placed in `workspace/agents/` is automatically
  available for use without any additional configuration steps.
- **SC-005**: Both OpenAI and Anthropic return equivalent-quality responses for
  the same task when configured with equivalent model capabilities.

## Assumptions

- Users have valid API keys for the providers they intend to use (OpenAI,
  Anthropic).
- Custom URLs point to OpenAI-compatible or Anthropic-compatible API endpoints
  (same request/response format).
- The workspace directory is located at the project root (`workspace/`) unless
  otherwise configured.
- Workspace directories follow the conventions established by Claude Code /
  OpenCode ecosystem for familiarity.
- The `memory/` subdirectory in workspace is reserved for session persistence
  (SQLite databases) — not for user-editable files.
- Formatting and linting configuration follows standard Python tooling (Black,
  Ruff, mypy) with pre-commit hooks, matching the project's existing
  conventions.
- Multi-model selection within a single provider (e.g., gpt-4 vs gpt-3.5) is
  handled at the profile level, not the provider configuration level.
